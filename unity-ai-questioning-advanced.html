1) 核心原则：让 AI 变成“加速器”

你负责方向，AI 负责加速。 你给目标、约束、上下文；AI 帮你更快写、改、查、复盘。

如果你只丢一句“帮我修一下”，AI 只能“猜”。你提供的信息越像 bug 报告 / 任务卡片，AI 越像靠谱队友。

三句“稳住不翻车”的指令（建议每次都带上）

  • 不要过度设计：只做当前需求，不要引入复杂架构/花哨模式。
  • 适合初级工程师理解：代码可读，变量命名清晰，带注释。
  • 先分析，再改代码：先列可能原因 + 验证步骤，再给最小修改方案。

2) 提问必填 8 个信息（不填就容易跑偏）

下面 8 条信息，能把“模糊提问”变成“工程对话”。你不一定都要填满，但越多越好：

  1. Unity 版本（例：2022.3 LTS）
  2. 平台（PC / Android / iOS / WebGL）
  3. 2D / 3D / URP / HDRP
  4. UI 方案（UGUI / UI Toolkit）
  5. 输入系统（旧 Input / New Input System）
  6. 目标（你想要的最终效果）
  7. 现象（目前发生了什么）
  8. 证据（报错全文、日志、相关代码、Inspector 配置截图文字说明）

4) 重构模板：能跑但乱 → 可维护

重构要避免“重写”。初级阶段最值钱的重构目标是：

  • 脚本职责更清晰（一个脚本一件事）
  • 少写重复代码（复制粘贴就是未来的 bug）
  • 更容易调试（关键逻辑能打点、能定位）

8) 万能检查清单：Unity 常见坑位

当你不知道从哪排查时，把下面清单贴给 AI，让它逐项带你验证：

事件 & 引用

  • 对象是否 active？脚本是否 enabled？
  • Inspector 引用是否丢失（Prefab override / 脚本改名）？
  • UI Button OnClick 是否绑定到 场景里那一个实例？（不是 Project 里的 Prefab）
  • Raycast 是否被挡（Canvas Group / Image Raycast Target / 层级遮挡）？

生命周期 & 时机

  • 初始化写在 Awake 还是 Start？对象启用顺序会影响吗？
  • 对象是否在运行时被 Destroy / SetActive(false)？
  • 协程是否被停止（StopAllCoroutines / 对象失活）？

物理 & 碰撞

  • Collider 是否勾选了 Is Trigger？Rigidbody 是否缺失？
  • Layer 碰撞矩阵是否允许？Tag/Layer 是否匹配？
  • 2D / 3D API 是否混用（OnTriggerEnter vs OnTriggerEnter2D）？

性能 & GC

  • Update 里是否有 Find / GetComponent / LINQ / 字符串拼接？
  • 是否频繁 Instantiate/Destroy（考虑对象池，但先确认真有问题）？
  • UI 刷新是否每帧重建（Layout Rebuild）？
